드디어 저희 스터디의 **문서화 작업을 위한 도큐사우루스**가 완성되었습니다!

정말 우여곡절이 많았지만, 그래도 생각보다 빨리 끝난거 같아 다행이네요.

아마 **마크다운** 형태의 블로그 작업을 이미 **벨로그**에서 해보셨을거기 때문에 큰 문제는 없겠지만, 해당 프로젝트를 활용하실 때 필요하다고 생각되는 내용들을 간단하게 정리보려고 합니다.

## 프로젝트 간단 설명 🔍

---

저희가 사용하게 될 해당 라이브러리는 **도큐사우루스**라는 정적 웹페이지 생성용 라이브러리입니다.

해당 라이브러리는 지금 작성된 [`mdx`](https://mdxjs.com/) 형태의 문서를 포함해서, **마크다운(`md`) 형식의 문서**를 활용하여 **문서(Docs)와 블로그(Blog)**를 작성하실 수 있습니다.

`md` 파일은 말그대로 마크다운 형식의 파일을 작성하는데 사용하는 확장자이고, `mdx` 는 살짝 생소하실 수도 있지만 **마크다운을 지원하면서 태그 혹은 컴포넌트, 즉 `jsx/tsx` 문법을 함께 사용할 수 있는 형태의 문서**라고 생각하시면 될 것 같습니다.

두 가지 중 원하시는 형태로 자유롭게 작성하시면 될 것 같습니다.

배포는 [Vercel](https://vercel.com/)을 통해 진행했습니다. 그래서 `clone` 을 진행하고 `main` 브랜치에 변화가 생길 때마다 재배포가 진행되어 작업하신 내용이 반영되니 편하게 작업하실 수 있으실겁니다.

패키지 관리는 **`npm` 으로 진행하고 있습니다.** <span style={{color: "#ff841f", fontWeight: "bold"}}>`yarn` 이 아닙니다!</span>

## 작업 방식 설명 🛠️

---

여타 프로젝트와 마찬가지로 **별도로 브랜치를 생성하여** 각자의 작업을 진행할 예정입니다.

브랜치 명과 관련된 형식은 다음과 같이 진행해주시면 감사하겠습니다.

```sh
git branch feature/[각자 이름]-[회차]

# 예시
git branch feature/samuel-1
```

별도의 브랜치로 작업하시는게 번거롭다고 생각이 드실 수도 있지만, 혹여나 발생할 수 있는 **`conflict`를 무시하고 덮어씌워 지는 일**이 없도록 하기 위함이기에 양해 부탁드리겠습니다. 별도의 `develop` 브랜치는 생성하지 않겠습니다. (각 브랜치에서 `main` 으로)

프로젝트 실행과 빌드는 다음 명령어로 가능합니다.

```sh
# 프로젝트 실행
npm start

# 프로젝트 빌드
npm run build
```

로컬에서 작업을 진행하실땐, `start` 명령어로 진행하시고 작업을 마무리하셨다면 `build` 명령어를 통해 <span style={{color: '#ff841f', fontWeight: 'bold'}}>꼭 빌드가 되는지 테스트를 진행하시고 `push` 와 머지를 진행하시면 될 것 같습니다.</span>

> 빌드가 되지 않는 경우, **Vercel에서 빌드가 되지 않는 오류가 발생할 수 있어** 잦은 머지보다는 빌드를 통해 기존에 확인 진행해주시고, 그래도 업데이트가 되지 않는다고 생각이 드신다면 저에게 별도로 연락주시면 감사하겠습니다.

그리고 우선적으로 진행하실 부분인 **Docs**는 `clone` 받으셨을 때의 `docs` 폴더 내 구조를 그대로 반영하고 있습니다. 그렇기에 해당 폴더 내에서 **각 도서마다 각자의 폴더를 만들어** 진행할 예정입니다.

그렇게 작업을 시작하시게 되면, `mdx` 혹은 `md` 형식의 파일으로 작성하시게 될텐데요 해당 포스팅과 관련된 데이터를 다음과 같이 작성해야합니다.

```mdx
---
slug: /test
title: Test입니다.
tags: [test1, test2]
sidebar_position: 1
---
```

기존 폴더를 확인하면 제 이름으로 되어있는 디렉토리와 안에 예제 포스팅인 `Test입니다` 파일이 존재하는 것을 확인하실 수 있으실 겁니다. 각 필드는 다음과 같은 의미를 갖습니다.

- `slug` : `url` 의 일부분을 의미하며 여기서는 **해당 포스팅의 경로**를 의미합니다.
- `title` : **사이드바와 포스팅 상단에 나타나는 제목**을 의미합니다. 설정하지 않을 경우 **파일명**을 기본값으로 갖습니다.
- `tags` : **포스팅과 관련된 태그를 포함하는 배열** 형태의 데이터입니다. 원하시는 태그를 여기에 추가하실 수 있습니다.
- `sidebar_position` : 말 그대로 **사이드바 내에서 갖는 순서**를 의미합니다. 설정하지 않을 경우 **오름차순**이 적용됩니다.

이정도의 데이터 입력만 한다면 충분하다고 생각되지만 추가적으로 작업하시고 싶으신 분들은 리서치를 통해 추가하시면 될 것 같습니다.

이후 문서 작성은 벨로그와 마찬가지로 **마크다운** 형식으로 작성하시면 될 것 같습니다!

**블로그** 탭의 데이터도 위와 마찬가지이며, 추가적으로 `authors` 라는 필드가 존재합니다. `authors` 에는 제가 기존에 셋팅한 `authors.yml` 파일의 이름을 배열에 추가하여 담으시면 됩니다.

이에 대한 내용은 이 포스팅의 소스코드 상단을 참고하시면 될 것 같습니다.

## 정보 수정에 관한 내용 🗂️

---

정보를 수정하시고 싶으실 때가 있으실거라고 생각합니다. 대부분의 정보 수정은 `docusaurus.config.js` 파일 내에서 가능합니다.

수정을 진행하시고 싶으시다면 이에 대한 내용 공유를 진행하고 공동의 동의 하에 진행해주시면 감사하겠습니다!

그리고 `authors` 에 대한 부분은 일단 임시로 제가 각자의 `github` 링크를 참조하여 구성해놓았습니다.

이 부분은 개인적으로 수정을 진행하셔도 좋으니 편하신대로 수정해주시면 감사하겠습니다.
